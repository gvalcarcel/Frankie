# Planificación técnica y pedagógica de Frankie Core v0.7.0

Fecha: 2026-06-27

Tipo de Work Order: `OFFLINE`

Estado: `alcance completado; v0.7.0 preparada para publicación`

## 1. Resumen ejecutivo

Frankie Core `v0.7.0` debe convertir la foundation publicada en `v0.6.0` en una herramienta con datos reutilizables, evidencias menos dependientes del texto libre y explicaciones más útiles para el aula.

El alcance recomendado mantiene el funcionamiento offline y de solo lectura. La versión no debe conectarse a Frankie, ejecutar comandos externos, administrar credenciales ni reparar sistemas.

El núcleo propuesto es:

1. Salida JSON para `status` y `audit`.
2. Contrato inicial de evidencias estructuradas y ejemplos saneados.
3. Doctor mejorado con urgencia y explicación pedagógica.
4. Documentación didáctica inicial para FP Básica.
5. Diseño formal de Live Mode, sin implementación real.

Esta selección aporta valor técnico y educativo sin convertir `v0.7.0` en una suma excesiva de funcionalidades.

Frankie es el servidor físico. VM100 es `srv-servicios` y VM101 es `srv-recursos`. Frankie Core es la herramienta software; no debe confundirse con el servidor ni con una VM.

Existe una implementación histórica bajo `cli/` con ideas de salida JSON y Markdown. No es la implementación activa de Frankie Core y no debe copiarse, borrarse o mezclarse sin una auditoría de migración específica.

## 3. Objetivos de v0.7.0

- Proporcionar una salida JSON estable para `status` y `audit`.
- Definir contratos de datos versionables y documentados.
- Introducir evidencias estructuradas como complemento verificable de Markdown.
- Mejorar Doctor como guía de diagnóstico y aprendizaje.
- Crear material inicial de aula para interpretar estado y auditoría.
- Diseñar la frontera de seguridad entre OFFLINE y LIVE.
- Mantener compatibilidad con la salida de consola de `v0.6.0`.
- Preservar el modo seguro por defecto y la ausencia de secretos.

## 4. Objetivos pedagógicos

| Propuesta | Aprendizaje esperado |
| --- | --- |
| Salida JSON | Distinguir datos estructurados de texto de presentación y reutilizarlos con otras herramientas. |
| Evidencias estructuradas | Comprender que una auditoría necesita fuente, fecha, resultado y trazabilidad. |
| Doctor mejorado | Separar problema, riesgo, impacto, urgencia y acción segura. |
| Documentación de aula | Leer una salida técnica, formular hipótesis y justificar conclusiones. |

| Recolectores futuros | Aprender a obtener evidencia sin alterar el sistema observado. |

El lenguaje dirigido al alumnado debe ser claro y preciso. La simplificación pedagógica no debe ocultar riesgos ni sustituir la terminología técnica que necesitan aprender.

## 5. Líneas de trabajo analizadas

### A. Salida JSON

**Decisión:** incluir JSON para `status` y `audit` en `v0.7.0`.

Valor técnico:

- separa datos de presentación;
- habilita automatización, tests, dashboard y API futuros;
- aprovecha los modelos tipados existentes (`StatusReport`, `AuditReport` y hallazgos);
- evita analizar texto de consola con expresiones frágiles.

Valor pedagógico:

- permite comparar el mismo resultado en consola y JSON;
- introduce objetos, listas, tipos, claves y serialización;
- facilita ejercicios con PowerShell, Python o `jq` en entornos compatibles.

Complejidad: media. El parser, el contrato de salida y la serialización deben ser únicos para evitar divergencia entre comandos.

Riesgos:

- crear claves inestables o dependientes del texto visible;
- duplicar lógica entre `console.py` y el serializador;
- reutilizar sin evaluación el renderer histórico de `cli/`;
- confundir salida JSON con escritura automática de ficheros.

Tests mínimos:

- JSON válido y determinista;
- esquema y tipos esperados;
- paridad semántica con consola;
- caracteres Unicode correctos;
- salida por `stdout` sin ficheros creados;
- compatibilidad de `--verbose` en `audit`;
- errores de argumentos con código de salida estable.

Documentación necesaria:

- contrato de claves y estados;
- ejemplos de `status --json` y `audit --json`;
- garantía de solo lectura;
- política de compatibilidad del esquema.

**Condicionado a margen:** JSON para `inventory` y `doctor`. Es deseable, pero debe entrar solo después de estabilizar el patrón común con dos comandos representativos.

### B. Evidencias estructuradas

**Decisión:** introducir el modelo documental y ejemplos saneados, sin automatizar todavía la captura live.

Propuesta de ubicación:

```text
docs/evidencias/structured/
├── README.md
├── schemas/
│   ├── status-evidence.schema.json
│   └── audit-evidence.schema.json
└── examples/
    ├── frankie-status.example.json
    ├── samba-validation.example.json
    └── portainer-warning.example.json
```

Campos comunes recomendados:

- `schema_version`;
- `evidence_id`;
- `captured_at` con zona horaria;
- `source_type`;
- `target`;
- `collector`;
- `status`;
- `summary`;
- `checks` o `observations`;
- `sanitization`;
- `source_references`.

Convivencia:

- Markdown sigue siendo la narración y evidencia histórica legible.
- JSON estructurado comienza como fuente complementaria.
- Ninguna evidencia estructurada sustituye una evidencia histórica de forma silenciosa.
- La prioridad entre fuentes debe ser explícita, basada en fecha, tipo y reglas documentadas.

Generación:

- en `v0.7.0`, ejemplos y evidencias podrán prepararse manualmente y validarse en tests;
- los recolectores automáticos se diseñarán aparte;
- la ejecución live se reservará para Work Orders LIVE.

Seguridad:

- lista de campos prohibidos: contraseñas, tokens, claves privadas y cookies;
- IPs, hostnames, usuarios y rutas deben sanearse según política de publicación;
- el campo `sanitization` debe declarar qué se eliminó o reemplazó;
- no se versionarán ficheros raw sin revisión humana.

### C. Doctor mejorado

**Decisión:** incluir una mejora acotada y compatible con el Audit Engine.

Cada diagnóstico debería distinguir:

- severidad técnica;
- nivel de urgencia (`informativa`, `planificable`, `prioritaria`, `inmediata`);
- riesgo;
- impacto probable;
- acción recomendada;
- pasos seguros;
- qué no hacer;
- evidencia que respalda el diagnóstico;
- explicación breve para alumnado.

Doctor debe seguir reutilizando Audit Engine en memoria. No debe ejecutar subprocess, conectarse a servidores ni convertirse en Repair Mode.

Valor pedagógico: permite trabajar el razonamiento diagnóstico como una secuencia: observar, interpretar, valorar impacto, proponer comprobaciones y decidir si se actúa.

### D. Preparación de Live Mode

**Decisión:** diseñar, no implementar.

Interfaz recomendada para estudiar:

```bash
python -m frankie status --live --profile <nombre>
python -m frankie audit --live --profile <nombre>
```

Se prefiere un modificador de los comandos existentes frente a comandos paralelos `live-status` y `live-audit`, porque mantiene una semántica común. La decisión definitiva requerirá una revisión arquitectónica.

**Decisión posterior:** el diseño final adopta comandos separados, `live-status` y `live-audit`, para hacer explícita la conexión y reducir activaciones accidentales. Siguen sin implementación. Véase `docs/frankie-core/live-mode-design.md`.

Requisitos del diseño:

- OFFLINE debe seguir siendo el valor predeterminado;
- `--live` debe exigir confirmación o configuración explícita;
- perfiles sin secretos dentro del repositorio;
- credenciales fuera de Git y gestionadas por mecanismos del sistema;
- operaciones permitidas mediante lista cerrada de solo lectura;
- timeouts, códigos de error y aislamiento por objetivo;
- registro de procedencia y fecha de la evidencia;
- saneamiento antes de guardar o publicar;
- prohibición explícita de repair y reinicios;
- pruebas con dobles antes de cualquier prueba real.

No se elegirá todavía SSH, agente local o API de Proxmox como mecanismo definitivo.

### E. Scripts seguros de recolección

**Decisión:** diseñar el contrato; implementar y probar en Work Orders posteriores.

Recolectores candidatos:

```text
scripts/collect_srv_servicios_readonly.sh
scripts/collect_srv_recursos_readonly.sh
scripts/collect_proxmox_readonly.sh
```

Principios:

- lista cerrada de comandos de lectura;
- `set -euo pipefail`;
- no instalar, reiniciar, habilitar, escribir configuración ni borrar;
- salida raw separada de salida saneada;
- timestamp UTC y objetivo explícito;
- inventario de campos sensibles eliminados;
- hash opcional para integridad;
- revisión humana antes de versionar;
- pruebas offline con fixtures antes de ejecutarlos en red.

Uso docente: comparar una captura raw privada con una evidencia saneada publicable y justificar cada eliminación.

### F. Perfil docente / alumnado

**Decisión:** no implementar flags `--teacher` o `--student` en `v0.7.0`.

La semántica técnica debe ser única. Dos perfiles de salida aumentarían combinaciones de tests y podrían generar explicaciones contradictorias.

En esta versión se recomienda documentación diferenciada en `docs/aula/`, manteniendo iguales los datos y estados técnicos.

### G. Documentación didáctica

**Decisión:** incluir un paquete inicial pequeño.

Documentos obligatorios propuestos:

```text
docs/aula/que-es-frankie.md
docs/aula/como-leer-un-status.md
docs/aula/como-leer-una-auditoria.md
docs/aula/actividad-status-json.md
docs/aula/actividad-audit-json.md
```

**Condicionado a margen:** `docs/aula/glosario-frankie.md` y actividades adicionales.

Cada actividad debe indicar objetivo, conocimientos previos, pasos, evidencias a entregar, preguntas de reflexión y criterios de evaluación.

### H. Carpeta histórica cli/

**Decisión:** mantener temporalmente y abrir una auditoría específica.

La carpeta contiene una implementación anterior con parser, payload y salida text/JSON/Markdown. La implementación activa está en el paquete raíz `frankie/` y presenta modelos y motores más evolucionados.

Una Work Order posterior debe:

- confirmar qué partes siguen siendo útiles;
- comparar contratos y tests;
- decidir si migrar patrones, archivar o eliminar;
- evitar imports ambiguos y duplicidad de documentación;
- verificar el impacto en empaquetado.

No se debe borrar ni reutilizar código de `cli/` durante la primera WO de JSON.

### I. Hallazgo Portainer 8000

**Decisión:** esperar a una Work Order LIVE.

`WO-LIVE-0001` debe comenzar con comprobaciones de solo lectura:

- configuración efectiva del contenedor;
- puertos publicados y procesos escuchando;
- uso real del puerto `8000`;
- reglas UFW aplicables;
- dependencia funcional de Edge Agent o túneles;
- evidencia saneada antes del cambio.

La decisión podrá ser mantener, cerrar o documentar el puerto. Cualquier cambio requerirá snapshot o backup adecuado, ventana de mantenimiento, rollback y pruebas de acceso a Portainer antes y después.

## 13. Criterios de aceptación de v0.7.0

### Funcionales

- `status --json` produce JSON válido y estable.
- `audit --json` produce JSON válido con checks, evidencias, estados y severidades.
- La salida de consola de `v0.6.0` sigue funcionando.
- Doctor muestra urgencia y explicación pedagógica sin reparar.
- Las evidencias estructuradas de ejemplo validan contra sus esquemas.

### Arquitectura

- Existe un único contrato de serialización para el paquete activo.
- Los comandos siguen siendo capas finas.
- Los renderers no recalculan lógica de negocio.
- Doctor sigue dependiendo de Audit Engine y Audit no depende de Doctor.
- `cli/` histórico no se importa desde `frankie/`.
- Los contratos quedan preparados para dashboard/API sin acoplarse a ellos.

### Seguridad

- El modo por defecto continúa siendo OFFLINE y solo lectura.
- No se requieren secretos para las funciones de `v0.7.0`.
- No se ejecutan comandos externos ni conexiones de red.
- No se escriben ficheros salvo una opción futura, explícita y separada que no forma parte del núcleo obligatorio.
- Fixtures y ejemplos no contienen datos sensibles.
- El hallazgo Portainer `8000` sigue visible hasta una WO LIVE.

### Calidad

- Todos los tests anteriores siguen pasando.
- Cada contrato JSON tiene tests positivos y negativos.
- Existen pruebas de compatibilidad y caracteres Unicode.
- Documentación y ejemplos coinciden con la implementación.
- La versión solo cambia durante la Work Order de release.

### Pedagogía

- Existen al menos cinco documentos iniciales de aula.
- Cada actividad define objetivo, pasos, evidencia y evaluación.
- Un alumno puede explicar la diferencia entre OFFLINE y LIVE.
- Un alumno puede identificar estado, evidencia, severidad, riesgo e impacto.
- El material ha sido revisado por el responsable docente.

## 14. Recomendación final

Avanzar con `v0.7.0` como una release de **datos estructurados y comprensión pedagógica**, no como una release de administración remota.

La primera implementación debe comenzar por auditar `cli/` y aprobar el contrato JSON. Solo después deben implementarse `status --json` y `audit --json`. Evidencias estructuradas, Doctor pedagógico y material de aula pueden desarrollarse en paralelo una vez establecidas las convenciones de datos.

Live Mode, Portainer y cualquier captura real deben permanecer en Work Orders separadas y explícitamente LIVE.

## Internal planning review

### 1. ¿La planificación puede realizarse sin Frankie en red?

Sí. Todo el análisis, los contratos, los esquemas, los tests y la documentación propuestos para esta planificación se basan en el repositorio local.

### 2. ¿Hay alguna propuesta que toque servidores?

No dentro de `v0.7.0`. Las capturas reales, las pruebas de recolectores y la revisión de Portainer se asignan a Work Orders LIVE futuras.

### 3. ¿Se han separado correctamente tareas offline y live?

Sí. La matriz clasifica diseño, implementación local y validación real por separado. La estrategia operativa establece OFFLINE como valor predeterminado.

### 4. ¿La propuesta es realista para el contexto de vacaciones?

Sí, siempre que se mantenga el núcleo obligatorio y el trabajo opcional no se convierta en requisito de release. Las tareas offline permiten avanzar sin coordinar acceso físico o ventanas de mantenimiento.

### 5. ¿El alcance de v0.7.0 es razonable?

Sí, con una condición: JSON para `inventory` y `doctor`, recolectores detallados y documentación ampliada deben permanecer opcionales.

### 6. ¿Hay riesgo de sobrecargar la versión?

Sí. Es el riesgo principal. Se mitiga con Work Orders pequeñas, criterios de aceptación por bloque y una revisión de alcance después de `status --json` y `audit --json`.

### 7. ¿La propuesta mantiene valor pedagógico?

Sí. Cada bloque técnico tiene un resultado de aprendizaje concreto y la documentación de aula forma parte del alcance obligatorio.
