# Planificación de Frankie Core v0.8.0

Fecha: 2026-06-28

Estado: ciclo abierto como `0.8.0-dev`

Tipo: `OFFLINE`

## 1. Objetivo de la versión

`v0.8.0` debe preparar a Frankie Core para trabajar con evidencias más completas y exportables, y acercar un futuro Live Mode de solo lectura sin activarlo de forma general ni introducir Repair Mode.

La versión debe crecer por contratos, simuladores y gates. Ningún bloque de planificación concede acceso a Frankie.

## 2. Contexto tras v0.7.0

`v0.7.0` está publicada y cerrada. Entregó:

- CLI offline completa;
- JSON para Status, Inventory, Audit y Doctor;
- evidencias estructuradas y comandos de consulta;
- Doctor pedagógico;
- documentación de aula;
- diseño de Live Mode;
- separación OFFLINE/LIVE.

Después de la release se incorporó un sistema de 24 agentes y seis plantillas de Work Orders. `v0.8.0-dev` será el primer ciclo que usa ese sistema desde su apertura.

## 3. Decisión de Product Owner

### Núcleo candidato de v0.8.0

1. Mejoras del sistema de evidencias.
2. Exportación controlada de informes.
3. Preparación avanzada de Live Mode con código desactivado y simuladores.
4. Limpieza técnica y consistencia del repositorio.
5. Actualización didáctica asociada a las capacidades anteriores.

### Trabajo LIVE complementario

1. Captura real de estado Frankie.
2. Revisión controlada de Portainer puerto 8000.
3. Validación avanzada de Samba.
4. Validación real de backups.

Las WOs LIVE no entran automáticamente en la release. Sus evidencias podrán alimentar el desarrollo si el usuario confirma acceso real y autoriza cada intervención.

### Diferido

- Dashboard funcional.
- API funcional.
- Repair Mode.
- IA aplicada.
- cambios de infraestructura.

Dashboard y API solo podrán recibir notas de arquitectura o contratos de salida si son necesarios para evitar decisiones incompatibles. No se implementarán en este ciclo sin una nueva decisión de alcance.

## 5. Bloques OFFLINE

### Evidencias v2

- campos de captura y frescura;
- validación más estricta;
- política de precedencia documentada;
- fixtures positivos y negativos;
- compatibilidad con las fichas v1.

### Exportación de informes

- salida a fichero solo mediante opción explícita;
- formatos inicialmente limitados a JSON y Markdown;
- rechazo de sobrescritura accidental;
- rutas locales controladas;
- tests de contenido y errores;
- sin datos live o sensibles por defecto.

### Live Mode preparado, no activo

- configuración con placeholders;
- interfaces de transporte desacopladas;
- simuladores y fixtures;
- allowlist y timeouts como contratos;
- feature gate desactivado;
- sin SSH, sockets ni conexión real en el flujo predeterminado.

### Limpieza técnica

- revisar la carpeta histórica `cli/`;
- detectar referencias obsoletas;
- consolidar estados de releases y evidencias actuales;
- mejorar tests de invariantes offline;
- no mezclar refactor con nuevas funciones.

### Aula

- actividad sobre frescura y precedencia de evidencias;
- actividad sobre exportación segura;
- explicación de simulación frente a captura live.

## 6. Bloques LIVE

Cada bloque exige una Work Order pequeña e independiente:

- captura real de host y VMs;
- inspección del puerto 8000 de Portainer;
- validación avanzada de permisos Samba;
- comprobación de backups y planificadores;
- revisión Proxmox solo si un objetivo concreto lo requiere.

Las tareas LIVE solo se ejecutarán cuando el usuario confirme acceso real a Frankie.

## 8. Riesgos

| Riesgo | Nivel | Mitigación |
|---|---|---|
| Confundir preparación con Live Mode activo | Alto | Feature gate, ayuda explícita y tests sin red |
| Escribir informes fuera de rutas esperadas | Medio | Opción explícita, validación y no sobrescritura |
| Romper evidencias v1 | Medio | Compatibilidad y fixtures de regresión |
| Publicar datos sensibles | Alto | Sanitización, Security Reviewer y evidencia raw privada |
| Sobrecargar v0.8.0 | Alto | Núcleo P0/P1 y dashboard/API diferidos |
| Mezclar limpieza y funcionalidad | Medio | WOs y commits separados |
| Depender de conectividad del aula | Medio | Camino OFFLINE independiente |

## 9. Dependencias

- modelos y serializadores actuales;
- loader de evidencias estructuradas;
- garantías de solo lectura;
- sistema de agentes y plantillas de WOs;
- fixtures saneados;
- decisión futura sobre transporte y credenciales para LIVE;
- autorización expresa del usuario para cualquier captura real.

## 10. Criterios de entrada

Un bloque puede empezar cuando:

- tiene una WO aprobada;
- agentes y propietario de decisión están asignados;
- alcance y fuera de alcance son explícitos;
- contratos y compatibilidad están identificados;
- tests y evidencia esperada están definidos;
- no depende de una acción LIVE no autorizada.

Para una WO LIVE se añaden identidad del objetivo, allowlist, timeout, saneamiento y criterios de parada.

## 11. Criterios de salida

`v0.8.0` podrá prepararse para release cuando:

- cada bloque incluido tiene auditoría y tests;
- los comandos anteriores siguen siendo compatibles;
- no existe conexión live implícita;
- evidencias v1 siguen siendo válidas o tienen migración documentada;
- exportación rechaza sobrescritura y datos inseguros;
- documentación técnica y de aula coincide con el comportamiento;
- todas las WOs LIVE incluidas tienen evidencia saneada y autorización;
- no existe Repair Mode;
- versión, changelog y notas están listos para auditoría de release.
