# Plan OFFLINE / LIVE de Frankie Core v0.8.0

Fecha: 2026-06-28

## 1. Regla principal

`v0.8.0-dev` comienza OFFLINE. Diseñar una capacidad LIVE no la activa ni autoriza una conexión.

**Las tareas LIVE solo se ejecutarán cuando el usuario confirme acceso real a Frankie.**

## 2. Camino OFFLINE

Puede avanzar sin Frankie en red:

- diseño de arquitectura y contratos;
- documentación y ADRs;
- refactor interno acotado;
- tests unitarios y de integración con fixtures;
- schemas y validadores;
- exportación local de informes;
- plantillas de configuración sin secretos;
- simuladores de transporte y respuestas;
- validación de allowlists y timeouts como datos;
- actividades pedagógicas;
- auditoría de repositorio.

### Reglas OFFLINE

- no SSH, socket, API remota o Docker SDK;
- no lectura de credenciales reales;
- no comandos sobre Docker, Samba o Proxmox reales;
- no escribir fuera de rutas temporales de test o destinos explícitos;
- no presentar fixtures como observaciones actuales;
- mantener `mode: offline` en salidas existentes.

### Entregables OFFLINE candidatos

| Entregable | Validación |
|---|---|
| Evidencias v2 | schemas, fixtures, compatibilidad y errores |
| Exportación | contenido, formato, rutas y no sobrescritura |
| Transporte simulado | cero red, timeouts simulados y allowlist |
| Limpieza técnica | tests de regresión y diff acotado |
| Aula | actividad guiada, resultado y rúbrica |

## 3. Camino LIVE

Solo con autorización expresa y conectividad confirmada:

- lectura real de servicios;
- captura de estado real del host y VMs;
- validación de Portainer;
- validación Samba avanzada;
- validación de backups;
- revisión Proxmox con objetivo concreto.

### Requisitos comunes LIVE

1. WO marcada `LIVE`.
2. `LIVE Operations Controller` asignado.
3. Objetivo y targets verificados.
4. Comandos permitidos y prohibidos.
5. Cuenta de mínimo privilegio.
6. Timeout y límite de salida.
7. Evidencia raw privada.
8. Política de saneamiento.
9. Criterios de parada.
10. Rollback y snapshot/backup si hay cambios.

### LIVE de solo lectura

Puede consultar estado y metadatos aprobados. No puede reiniciar, instalar, editar, borrar, cambiar permisos o modificar configuración.

### LIVE con cambios

Queda fuera del núcleo de `v0.8.0`. Una observación puede recomendar otra WO, pero nunca transformarse automáticamente en cambio.

## 4. Frontera técnica propuesta

```text
Core domain
    ├── Offline evidence provider (activo)
    └── Live provider interface (diseño / gate desactivado)
            ├── simulator (tests)
            └── real transport (no disponible por defecto)
```

El dominio no debe depender de SSH, shell o una API concreta. Un adaptador real futuro deberá estar aislado y bloqueado salvo configuración y autorización válidas.

## 5. Evidencia y saneamiento

- Raw LIVE: privada, temporal y no versionable por defecto.
- Saneada: elimina secretos, credenciales, IPs internas y datos personales.
- Publicable: incluye fecha, objetivo, método, origen lógico y declaración de saneamiento.
- Fixture: se identifica como simulación y nunca como captura real.

## 6. Matriz de decisión

| Pregunta | Si la respuesta es sí | Clasificación |
|---|---|---|
| ¿Conecta a Frankie o una VM? | Requiere autorización | LIVE |
| ¿Consulta un servicio real? | Requiere allowlist | LIVE |
| ¿Usa solo fixtures locales? | Puede continuar sin red | OFFLINE |
| ¿Modifica configuración o datos? | WO separada, backup y rollback | LIVE con cambios |
| ¿Solo diseña interfaces? | No implementar transporte real | OFFLINE |

## 7. Criterios de parada LIVE

Detenerse si:

- falta autorización;
- la identidad no coincide;
- el comando no está permitido;
- se requiere más privilegio;
- aparece información sensible inesperada;
- cambia la conectividad;
- una consulta produce efectos laterales;
- la evidencia contradice una condición crítica;
- no puede sanearse la salida con seguridad.

## 8. Relación con la release

El camino OFFLINE puede cerrar una release sin ejecutar todas las WOs LIVE. Si una evidencia LIVE se declara requisito de release, deberá indicarse explícitamente en sus criterios de salida.

Portainer, Samba y backups siguen en WOs LIVE separadas. Repair Mode continúa fuera de alcance.
