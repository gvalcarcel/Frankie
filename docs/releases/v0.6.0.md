# v0.6.0 - Frankie Core / CLI Foundation

Fecha: 2026-06-26

Estado: preparada localmente, pendiente de auditoría final, tag y publicación en GitHub.

## 1. Resumen ejecutivo

`v0.6.0` prepara la primera versión estable de Frankie Core como núcleo Python de la plataforma Frankie.

Esta release convierte la CLI en una herramienta de consulta, inventario, auditoría y diagnóstico basada en evidencias documentadas del repositorio. No administra servidores en vivo, no repara nada y no ejecuta comandos externos contra la infraestructura.

El objetivo principal es ofrecer una base estable, pedagógica y ampliable para futuras fases de monitorización, evidencias estructuradas, live mode, dashboard e integración con IA.

## 2. Qué incluye

- Paquete Python raíz `frankie/`.
- CLI ejecutable con `python -m frankie`.
- Comando `version`.
- Comando `help`.
- Comando `status`.
- Comando `inventory`.
- Comando `audit`.
- Comando `audit --verbose`.
- Comando `doctor`.
- Comando `doctor --verbose`.
- Audit Engine v1 con checks documentales.
- Doctor Engine MVP basado en hallazgos del Audit Engine.
- Tests automatizados de los comandos foundation.
- Documentación de Frankie Core.
- Evidencias de readiness y validación pre-release.
- Sincronización SMB con evidencia pre-release validada.

## 6. Riesgos conocidos

| Riesgo | Estado | Tratamiento |
| --- | --- | --- |
| Portainer publica el puerto `8000`. | `WARN` | Mantener visible hasta decisión técnica futura. |
| Frankie Core se basa en evidencias documentadas. | Aceptado | Evolucionar hacia evidencias estructuradas. |
| No hay conexión live. | Aceptado | Diseñar en versión futura con permisos y alcance claros. |
| No hay modo repair. | Aceptado | Mantener seguridad y control humano. |

## 7. Limitaciones

Esta release no implementa conexión live con Frankie físico.

Esta release no repara nada.

Esta release no ejecuta comandos externos contra servidores.

Esta release se basa en evidencias documentadas.

Portainer puerto `8000` sigue documentado como `WARN`.

SMB queda validado por evidencia pre-release.

## 8. Qué no incluye todavía

- Salida `--json`.
- Evidencias estructuradas.
- Live status.
- Live audit.
- Conexión SSH.
- Integración directa con Docker, Samba o systemd.
- Repair mode.
- Dashboard.
- Monitorización en tiempo real.
- Integración IA.

## 9. Uso pedagógico

Esta release permite explicar al alumnado:

- qué diferencia hay entre una versión `dev` y una versión estable;
- por qué una release necesita pruebas;
- qué es un changelog;
- qué es un tag;
- qué es una release;
- por qué se separa preparación de publicación;
- por qué un diagnóstico seguro no debe reparar automáticamente;
- cómo se documentan riesgos conocidos antes de publicar.

## 10. Criterios de validación

Antes de publicar esta release deben cumplirse:

- `python -m frankie version` muestra `0.6.0`;
- todos los comandos foundation funcionan;
- `python -m unittest discover -s tests` pasa;
- `python -m compileall frankie` pasa;
- no hay secretos versionados;
- no hay cachés Python;
- no hay tag creado durante la preparación;
- no hay release creada durante la preparación;
- no hay push durante la preparación.
